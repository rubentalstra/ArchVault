# Phase 5a — Community Registry

## Status: Not Started

## Goal

Integrate a community block registry hosted on GitHub, allowing the community to contribute and share blocks.

## Prerequisites

- Phase 4c (Block Browser & Install) — complete

## Architecture

The community registry is a **public GitHub repository** (`archvault/block-registry`) containing:

```
archvault/block-registry/
├── index.json              ← manifest of all blocks (auto-generated by CI)
├── blocks/
│   ├── aws-lambda-stack/
│   │   ├── block.json      ← the actual block file
│   │   └── README.md       ← description, screenshots
│   ├── kubernetes-microservices/
│   │   ├── block.json
│   │   └── README.md
│   └── ...
└── .github/
    └── workflows/
        └── validate.yml    ← CI validates blocks on PR
```

Contributors submit blocks via Pull Requests. CI validates the block JSON against the schema. Maintainers review
and merge. The app fetches `index.json` to discover available community blocks.

## Data Model

**`community_block_cache`** (server-side cache of the registry index):

| Column       | Type      | Notes                                    |
|--------------|-----------|------------------------------------------|
| `id`         | uuid, PK  |                                          |
| `index_json` | jsonb     | Cached content of index.json             |
| `etag`       | text      | GitHub API ETag for conditional requests |
| `fetched_at` | timestamp | When the cache was last refreshed        |

This is a single-row cache table. Refreshed every 15 minutes (or on user request).

## Tasks

### Registry Repository Setup

- [ ] Create `archvault/block-registry` repo structure (README, contributing guide, example block)
- [ ] GitHub Actions workflow `validate.yml`:
    - Triggered on PR to `main`
    - Finds all `blocks/*/block.json` files
    - Validates each against the Zod block schema (via a small Node script)
    - Reports validation errors as PR comments
- [ ] GitHub Actions workflow `build-index.yml`:
    - Triggered on push to `main`
    - Generates `index.json` from all valid blocks (metadata only, not full block content)
    - Commits `index.json` back to the repo

### `index.json` Format

```jsonc
{
  "generated_at": "2026-03-14T12:00:00Z",
  "blocks": [
    {
      "id": "aws-lambda-stack",          // directory name
      "name": "AWS Lambda Stack",
      "summary": "Serverless API with Lambda + DynamoDB",
      "block_type": "container",
      "category": "cloud",
      "tags": ["aws", "serverless"],
      "author": "contributor-name",
      "version": "1.0.0",
      "icon": "cloud",
      "path": "blocks/aws-lambda-stack/block.json"  // relative path for fetching
    }
  ]
}
```

### Server-Side Integration

- [ ] Drizzle schema for `community_block_cache` table
- [ ] Server function `fetchRegistryIndex()`:
    - Check cache age (< 15 minutes → return cached)
    - Fetch `index.json` from GitHub raw URL with `If-None-Match` header (ETag)
    - If 304 Not Modified → return cached
    - If 200 → validate, update cache, return
    - If fetch fails → return stale cache with warning
- [ ] Server function `fetchCommunityBlock(path)`:
    - Fetch individual `block.json` from GitHub raw URL
    - Validate against Zod schema before returning (security!)
    - Cache in memory (short TTL, not in DB)
- [ ] Rate limiting: max 60 GitHub API requests per hour (respect unauthenticated limit)
    - Use conditional requests (ETag) to minimize actual fetches

- [ ] All community block JSON is validated against the Zod schema before install

- [ ] Community blocks cannot contain: scripts, HTML, URLs with javascript:, or any executable content
