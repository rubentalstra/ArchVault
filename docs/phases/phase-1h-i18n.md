# Phase 1h — Internationalization (i18n)

## Status: Not Started

## Goal

Add multi-language support to Archvault using TanStack Router's URL rewriting for locale-aware routing and Paraglide JS
for compile-time type-safe translations.

## Prerequisites

- Phase 1a (Scaffold) — project structure exists
- Phase 1c (Authentication) — routes exist to localize

## Context

TanStack Router does not ship a built-in i18n system. Instead, it provides **URL rewriting** primitives (`input`/
`output` on the router) that strip and add locale prefixes during routing. The actual translations come from a separate
library.

**Paraglide JS** (by inlang) is the recommended choice because:

- Official TanStack Start examples exist in the TanStack Router repo
- Compile-time type safety — translation keys and parameters are fully typed
- Zero runtime overhead — unused translations are tree-shaken
- Vite plugin integration is straightforward
- Aligns with the project's strict TypeScript policy

### Routing Strategy

- **No prefix for default locale**: `/about` (English)
- **Prefix for other locales**: `/nl/about`, `/de/about`
- Public pages use URL prefixes for SEO
- Authenticated pages (`/dashboard`, `/admin`) use cookie-based locale (no URL prefix — SEO irrelevant)

## Implementation Steps

### Step 1: Paraglide Project Setup

Create `project.inlang/settings.json`:

```json
{
  "$schema": "https://inlang.com/schema/project-settings",
  "sourceLanguageTag": "en",
  "languageTags": [
    "en",
    "nl"
  ],
  "modules": [
    "https://cdn.jsdelivr.net/npm/@inlang/message-lint-rule-empty-pattern@latest/dist/index.js",
    "https://cdn.jsdelivr.net/npm/@inlang/message-lint-rule-missing-translation@latest/dist/index.js",
    "https://cdn.jsdelivr.net/npm/@inlang/message-lint-rule-without-source@latest/dist/index.js"
  ],
  "plugin.inlang.messageFormat": {
    "pathPattern": "./messages/{languageTag}.json"
  }
}
```

### Step 3: Vite Plugin Configuration

Add to `vite.config.ts`:

```ts
import {paraglideVitePlugin} from "@inlang/paraglide-js";

export default defineConfig({
    plugins: [
        paraglideVitePlugin({
            project: "./project.inlang",
            outdir: "./src/paraglide",
            outputStructure: "message-modules",
            cookieName: "ARCHVAULT_LOCALE",
            strategy: ["cookie", "preferredLanguage", "baseLocale"],
        }),
        // ... existing plugins
    ],
});
```

### Step 4: Locale Utilities

Create `src/lib/i18n.ts`:

```ts
export const DEFAULT_LOCALE = "en";
export const SUPPORTED_LOCALES = ["en", "nl"] as const;
export type Locale = (typeof SUPPORTED_LOCALES)[number];
export const LOCALE_COOKIE = "ARCHVAULT_LOCALE";

// Paths that skip locale prefix handling (API, auth, admin dashboard)
export const IGNORED_PATHS_REGEX = /^\/(?:api|_protected)(?:\/|$)/;

export function isValidLocale(locale: string): locale is Locale {
    return SUPPORTED_LOCALES.includes(locale as Locale);
}

export function deLocalizeUrl(url: URL): URL {
    const segments = url.pathname.split("/").filter(Boolean);
    if (segments.length > 0 && isValidLocale(segments[0])) {
        // Redirect /en/... to /... (default locale should have no prefix)
        if (segments[0] === DEFAULT_LOCALE) {
            url.pathname = "/" + segments.slice(1).join("/");
            return url;
        }
        // Strip locale prefix for routing
        url.pathname = "/" + segments.slice(1).join("/");
    }
    return url;
}

export function localizeUrl(url: URL, locale: Locale): URL {
    if (locale === DEFAULT_LOCALE) return url; // no prefix for default
    if (IGNORED_PATHS_REGEX.test(url.pathname)) return url;
    url.pathname = `/${locale}${url.pathname}`;
    return url;
}
```

### Step 5: Router URL Rewriting

Add the `rewrite` option to the router configuration:

```ts
const router = createTanStackRouter({
    routeTree,
    rewrite: {
        input: ({url}) => deLocalizeUrl(url),
        output: ({url}) => localizeUrl(url, currentLocale),
    },
});
```

### Step 6: Dynamic `<html lang>`

Update `src/routes/__root.tsx` to set `<html lang={locale}>` based on the current locale (from cookie or URL).

### Step 7: Use Translations in Components

```tsx
import {m} from "#/paraglide/messages";

function MyComponent() {
    return <h1>{m.app_name()}</h1>;
}
```

All keys are type-checked — referencing a non-existent key or missing a required parameter is a compile-time error.

## Locale Detection Priority

1. **URL path prefix** (for public pages): `/nl/about` → Dutch
2. **Cookie** (`ARCHVAULT_LOCALE`): for authenticated pages and persistence
3. **Browser `Accept-Language` header**: SSR fallback
4. **Default locale** (`en`): ultimate fallback

---

## Best Practices

- Public pages use URL prefixes (`/nl/about`) for SEO
- Authenticated areas (`/dashboard`, `/admin`, `/settings`) use cookie-based locale only — no URL prefix
- Default locale has no prefix: `/about` not `/en/about`
- Redirect default-locale prefixed URLs: `/en/about` → 301 → `/about`
- Set `<html lang={locale}>` for accessibility
- Start with a small set of locales (en, nl) and add more incrementally
- Translation files are flat JSON — avoid deep nesting

---

## Adding a New Locale

1. Add language tag to `project.inlang/settings.json` → `languageTags`
2. Add language tag to `SUPPORTED_LOCALES` in `src/lib/i18n.ts`
3. Create `messages/{locale}.json` with translations
4. Paraglide auto-generates typed accessors on rebuild
